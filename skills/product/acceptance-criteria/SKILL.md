# Skill: Acceptance Criteria

## When to use this skill

When filling the third column of a user stories table, or when reviewing AC for
clarity and completeness.

## Three valid AC formats in this team

### Format 1: Plain condition bullets (most common)

For new stories where Jira tickets don't yet exist:

```
- AS+ add-on is bought and active for the Location
- FF is ON
- Listing capability for DA is purchased and expiration date is still valid
```

Each bullet is an observable condition. No verbs like "should", "must", "will" —
just the state that needs to be true.

### Format 2: Ticket-linked AC (for in-flight or completed work)

When AC references implementation tickets (after the PRD has been broken down):

```
- LM-3865: Add feature flag for Data Axle (RELEASED)
- LM-3914: Add DataAxle availability check based on FF addon (RESOLVED)
- LM-3915: Hide/Show UI related to DataAxle based on isDataAxleAvailable parameter (RESOLVED)
```

Status badges (RELEASED, RESOLVED, IN TESTING, OPEN) are added by humans in
Confluence, not generated by the agent.

NEVER invent Jira ticket numbers. If you don't have them, use Format 1.

### Format 3: Mixed (rare, for complex stories)

```
- if Location contains GBP related data, we should submit it to DataAxle
- if no GBP data, submit just core business data
- Document what fields may be submitted in both cases
- show validation errors (REQUIRED SEPARATE DISCUSSION)
```

The "REQUIRED SEPARATE DISCUSSION" tag is a real pattern from this team — it
flags AC that need their own spec session. Use sparingly.

## What makes AC observable

Good AC are testable. Someone can look at the system and say "yes, this is true"
or "no, it's not".

### Bad (subjective)
- The flow feels intuitive
- Users understand the value
- Performance is acceptable

### Good (observable)
- Bundle option appears when user has 2+ aggregators in cart
- "Connect & Sync" module heading is visible above the integration list
- API response time for connection check is under 500ms p95

### Borderline (requires explicit definition)
- "Active Sync Plus is required" — only observable if "required" is defined
  somewhere (e.g., paywall component shown, button disabled with message).

## Common pitfalls

### Pitfall 1: Restating the user story

Bad:
```
| User story | Acceptance criteria |
|---|---|
| As a Customer, I can buy multiple aggregators | - User can buy multiple aggregators |
```

The AC just rephrases the story. It adds no information. Don't do this.

### Pitfall 2: Implementation details masquerading as AC

Bad:
```
- Use Braintree subscription API
- Add migration for new aggregator_purchase table
```

These are tasks, not acceptance criteria. AC describes user-facing or
behavior-facing conditions, not how to implement them.

### Pitfall 3: AC that depend on hidden state

Bad:
```
- The system handles the case correctly
```

What case? Correctly how? AC must be self-explanatory to someone who has read
the user story but not your private notes.

### Pitfall 4: Counting AC

If a story has 1 AC, it's probably under-specified.
If a story has 10+ AC, it's probably over-decomposed — split into two stories.

Sweet spot: 2-5 AC per story.

## Writing AC for different story types

### Behavior change AC

For stories that change what the user sees or does:
- "User sees [X] when [condition]"
- "[Action] triggers [outcome]"
- "[Element] is visible/hidden when [state]"

### State condition AC

For stories about access, permissions, gating:
- "[Capability] is active for the Location"
- "[Add-on] is purchased and not expired"
- "[Feature flag] is ON"

### Validation AC

For stories about error handling and edge cases:
- "Validation error is shown when [bad input]"
- "Save is blocked until [precondition]"
- "Empty state is shown when [no data]"

### Investigation/Spike AC

For investigation user stories (e.g., "As a CB engineer, I want to submit
location data to Data Axle via an internal API"):
- "API endpoint exists at [path]"
- "Endpoint accepts [payload structure]"
- "Submission status can be retrieved by [identifier]"
- "Documented learnings include [list of questions answered]"

## When AC needs a separate discussion

Sometimes a story is ready to commit but specific AC need their own spec
session. The team uses this marker:

```
- show validation errors (REQUIRED SEPARATE DISCUSSION)
```

Use this when:
- The AC is genuinely needed but its specifics depend on design/research not yet done
- You don't want to block the story by waiting for that detail
- You don't want to invent the detail and risk it being wrong

## Self-check before finishing

For each story's AC, ask:
- [ ] Can someone test/verify this without asking me follow-up questions?
- [ ] Does the AC add information beyond the user story?
- [ ] Are there 2-5 AC bullets (not 1, not 10+)?
- [ ] No invented Jira ticket numbers?
- [ ] No "should", "must", "will" — just observable state/behavior?
- [ ] No words like "intuitive", "seamless", "robust"?
